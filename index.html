01 — The Problem

Drowning in
Scanned Legal Paper

Court systems deal with massive scanned PDFs — non-searchable, bilingual, irregular layouts — that lock down GPU pipelines for 15 minutes per file.

📄
Non-Searchable Scans
Hundreds of image scans with varying rotations, poor print quality, and mixed font types.
🌐
Bilingual Complexity
English and Hindi legal text interleaved on the same page, needing dual-engine fusion.
⏱
GPU Bottlenecks
Traditional visual models take 15 minutes per file, stalling hardware resource queues.
🗂
Manual Processing
Staff waste hours classifying Vakalatnamas, Affidavits, Petitions, and Annexures manually.
02 — Architecture

Four Stages.
High-Speed Pipeline.

A parallelized CPU/GPU orchestrator that combines ultra-fast OCR with surgical VLM usage.

01
Index Extraction
30 – 60 sec
Locate the Index Sheet in pages 1–10 with PaddleOCR, then parse it instantly to JSON via Qwen2.5-VL.
02
Bilingual OCR
1.5 – 2 min
Scan remaining pages in parallel. Tesseract fallback resolves any low-confidence Devanagari lines.
03
Vector Indexing
8 – 10 sec
Convert legal chunks into 1024-dim dense vectors using BGE-M3, then store instantly in Qdrant DB.
04
Verification
12 sec
Cross-check extracted indices against actual page ranges and flag any gaps for lawyer review.

⚡ Fast Hybrid Mode
CPU / VLM Parallel
CPU handles bulk detection across 10+ threads. The VLM fires exactly once — on the confirmed index page only. Ideal for standard petition bundles.
  • Pages 1–10 scanned in parallel at 220 DPI
  • Score ≥ 4.0 triggers VLM parse
  • Column fallback if no index found
  • Best for standard petition bundles
🎯 Strict VLM Mode
Full Visual Parse
Bypasses all heuristics. Qwen2.5-VL processes each page directly from its rendered image — no OCR pre-processing required.
  • Direct image → structured JSON
  • Handles complex multi-column layouts
  • Best for hand-stamped irregular scans
  • Higher accuracy, more compute
